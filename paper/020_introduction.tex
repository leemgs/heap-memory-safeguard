\section{Introduction}
Android applications often allocate through two distinct paths.  ART manages Java
and Kotlin objects, while JNI libraries, media pipelines, graphics engines, and
inference runtimes allocate native memory.  Linux ultimately accounts the resident
pages and exposes pressure through memory cgroups and Pressure Stall Information
(PSI).  Native allocation, virtual mapping, page commitment, RSS growth, memcg
charging, and page faults are separate events; consequently, it is inaccurate to
say that ART is universally unaware of native memory until a page fault.  Android
also provides native-allocation notification paths.  The narrower problem studied
here is that managed-heap telemetry, sampled native-allocation activity, and
page-level pressure are not presented as one per-application control state.

We call this incomplete combined view a \emph{visibility gap}.  We call the
independent timing of runtime collection and kernel reclaim a \emph{temporal gap}.
These are design hypotheses, not conclusions from a device study in this paper.
Linux and Android already provide capable actuators, including LMKD/PSI policies,
memory-cgroup reclaim, proactive reclaim, DAMON/DAMOS, MGLRU, and application
compaction.  HMS therefore does not claim a new reclamation primitive.  Its proposed
contribution is a small coordination policy that selects when and how much of an
existing per-cgroup actuator should be requested.

The repository accompanying this paper contains a deterministic Python simulation.
It does not contain an AOSP framework patch, ART hook, kernel patch, SELinux policy,
commercial-device trace, or per-run physical measurement.  Earlier drafts
incorrectly described simulated values as commercial-device measurements.  This
revision removes those claims and limits every conclusion to the artifact that can
be audited.

\paragraph{Contributions.}
This paper makes three bounded contributions:
\begin{enumerate}
  \item It specifies a per-application state model that separates managed
  allocation, sampled native allocation, resident occupancy, and kernel pressure.
  \item It defines a two-threshold controller with online-only normalization,
  bounded reclaim requests, cooldown, and an explicit multi-process fairness rule.
  \item It provides a seeded simulation and scripts for replaying four workload
  families and sweeping controller parameters.  The simulation is useful for
  hypothesis generation; it is not evidence of production performance.
\end{enumerate}
